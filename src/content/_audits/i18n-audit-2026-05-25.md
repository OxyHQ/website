# i18n Coverage Audit — 2026-05-25

## Summary

- **Total packages/apps audited:** 18 (6 OxyHQServices packages, 6 consumer apps, 1 shared UI library, 1 docs/website, plus tnp, OxyOS shell, Astro browser, Bloom).
- **Apps with working i18n infrastructure:** 9 (accounts, inbox, services, Mention, Allo, Homiio, Alia, Clarity, tnp).
- **Apps shipping multilingual content (>1 non-English locale populated with >50% coverage):** 4 (Mention, Allo, Homiio, tnp).
- **Apps where Arabic is declared but unpopulated:** 2 (`@oxyhq/core`, `@oxyhq/inbox`). Catalan, French, German, Italian, Japanese, Korean, Portuguese, Chinese: same situation in both — registered as locale codes but file body is `export default en`.
- **RTL support:** zero apps in this ecosystem call `I18nManager.allowRTL(true)` / `forceRTL(true)`. `@oxyhq/services` *reads* `I18nManager.getConstants().isRTL` in TextField + Icon + AnimatedText but never enables RTL. Arabic users would see LTR layout.
- **No-i18n packages that are user-facing:** `@oxyhq/auth` (web auth IdP), `@oxyhq/console` (Cloud console), `oxy-website`, `Bloom`. Hundreds of hardcoded English strings each.

### Critical gaps

1. **`@oxyhq/auth`** — Entire login / signup / 2FA / authorize / settings flow is English-only. This is the IdP every Oxy user passes through. Highest user impact.
2. **`@oxyhq/console`** — Entire Cloud dashboard (billing, usage, apps, keys, docs) is English-only.
3. **`@oxyhq/core` non-EN/ES locales** — Files exist for 9 additional locales but contain only 79 of 968 keys (8.2 % coverage). Apps that fall back to core (accounts, inbox) silently surface raw keys to ca/fr/de/it/ja/ko/pt/zh/ar users.
4. **`@oxyhq/inbox` non-EN/ES locales** — Same shape: 9 placeholder files that `export default en` and a TODO comment about RTL never wired up.
5. **`oxy-website`** — Marketing + docs site is English-only; no `astro-i18next` / equivalent setup. This is the public face of Oxy.

### `@oxyhq/core` (packages/core)

- **Library:** Custom `translate(locale, key, vars)` function reading bundled JSON dictionaries.
- **Locales (11 files):** ar-SA, ca-ES, de-DE, en-US, es-ES, fr-FR, it-IT, ja-JP, ko-KR, pt-PT, zh-CN.
- **Coverage:**
  - en-US: 100 % (canonical, **968 keys**)
  - es-ES: 100 % (968 keys)
  - ar-SA, ca-ES, de-DE, fr-FR, it-IT, ja-JP, ko-KR, pt-PT, zh-CN: **79 keys each → 8.2 % coverage**. All 9 are effectively stubs covering only auth (`signin.*`, `signup.*`) basics; everything else falls through to the raw key.
- **RTL:** N/A (pure logic package), but no helper exported either (no `isRTL(locale)`, no RTL stylesheet helpers). Apps must implement their own.

## Recommendations

1. **Highest priority: ship i18n for `@oxyhq/auth`.** Add `i18next` + `react-i18next` (consistent with Mention/Allo/Homiio/tnp), populate en + es + the EU set (ca/fr/de/it/pt) at minimum. Every Oxy user passes through this IdP, and it's currently English-only.

2. **Second priority: close the `@oxyhq/core` 9-locale gap.** Translate the missing **889 keys × 9 locales** in `packages/core/src/i18n/locales/{ar,ca,de,fr,it,ja,ko,pt,zh}.json`. This single fix unlocks correct strings for `@oxyhq/services`, `@oxyhq/accounts`, and any other consumer that relies on `translate()`. Recommend feeding the en + es files to a translation provider as 18 batches (9 languages × 2 chunks) — they're ordinary JSON, fully isomorphic.

3. **Third priority: populate `@oxyhq/inbox`'s 9 placeholder locales.** Same shape as core but TS-based — copy `en.ts` to each target and translate values. Add a settings → language route in the inbox app to expose `setLocale` (infra exists, UI doesn't).

4. **Wire RTL once and broadcast it.** Add a single helper to `@oxyhq/core` (e.g. `isRTL(locale: Locale): boolean` returning `true` for `ar-SA` / `he-IL`) and have each app's root layout call `I18nManager.allowRTL(true)` + `I18nManager.forceRTL(isRTL(currentLocale))`. Without this, the `isRTL` reads in `TextFieldFlat`/`TextFieldOutlined`/`Icon`/`AnimatedText` are dead code for Arabic users.

5. **Add i18n to `oxy-website`.** For a React + Vite site, `react-i18next` is the consistent choice (matches Mention/Allo/Homiio/tnp). Start with en + es (matching what the rest of the ecosystem actually translates fully).

6. **Translate the hardcoded literals already breaking otherwise-localized apps.** Roughly 30–50 high-traffic strings to fix first, ranked by user-visible frequency:
   - `@oxyhq/services` `FeedbackScreen` (Back / Next / Type / Category / Priority / Title / Contact)
   - `@oxyhq/services` `OxyAuthScreen` (Sign in with Oxy / Cancel)
   - `@oxyhq/services` `CreateManagedAccountScreen` (Username / Display Name / Bio / Create Identity)
   - `@oxyhq/inbox` `MessageDetail` (Mark unread / Report spam / Reply all)
   - `@oxyhq/inbox` `AccountSwitcher` (Add another account / Settings / Sign out)
   - `@oxyhq/inbox` `ComposeForm` (Send now / Schedule send)
   - `@oxyhq/accounts` `error-fallback` (Something went wrong / Try again / Error details)
   - `Mention` `SignInPrompt`, `WelcomeModal`
   - `Allo` `SignInPrompt` (Welcome to allo + feature bullets)
   - `Homiio` `sindi` (Know your rights / Legal guidance / Community support)

7. **Reconcile Mention's locale key counts.** es-ES (859) and it-IT (854) have more keys than en-US (827) — implies en is missing keys present in translations. Run a key-diff and merge missing keys back into en.json so the canonical dictionary is exhaustive.

8. **Console-as-developer-surface:** acceptable to keep English-only for now, but at least extract strings into a constants file so a future i18n migration isn't a 70-file diff.

9. **Bloom:** keep its no-i18n posture. UI primitives should remain string-free.

10. **Astro browser:** out of scope — defer to upstream Chromium i18n.
